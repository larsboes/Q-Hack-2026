2. KEY DECISIONS
═══════════════════════════════════════════════════════════════════════════════

a) main.ts exceeds 80 lines (~115 lines)
   Reason: The spec called for settings tab (PluginSettingTab) which requires
   its own class, plus two command handlers, plus the offline fallback in each.
   Splitting into two files would require a build step; single file is simpler
   for a demo. checkpoint-2/main.ts is kept under 50 lines as a "before" state.

b) Mock responses are written as realistic terminal output
   The plan-output.txt and agent-answer.txt reproduce OpenCode's actual output
   format (version line, model line, boxed plan, token counts). This makes the
   mock mode feel authentic rather than obviously fake.

c) checkpoint-3 shows agent improvement beyond the prompt
   The fixed calculator.ts adds a zero-division guard that wasn't asked for.
   This is intentional — it demonstrates "the agent going slightly beyond the
   brief" which is a talking point in rehearsal-2-cli.md.

d) run-demo.sh uses `grep` for display (not cat + full file)
   Only shows the relevant divide() block in Step 1 to keep terminal output
   focused. Full file shown via cat in later steps.

e) CLI fallback opencode.json uses permission strings not objects
   The spec showed `"bash": "ask"` (string) not `"bash": { "*": "ask" }` (object).
   This matches the simpler permission form valid in OpenCode config.
